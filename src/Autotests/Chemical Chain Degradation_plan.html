Test Chemical Chain Degradation

Overall status be found in the status document

More information on the requirements, funtionality, and testing can be found at FRAMES Wiki Wiki Documentation Site: Task 5 Chemical Chain Degredation

Testing Stragety

1. Subtask A: Chemical decay in 11 MEPAS modules
Then each component will be tested within the system using automated tests which run a case to check that both radionuclide and chemical decay processes are behaving properly. First the case will be run by hand and hand checked. Then the case will be automated to test that future versions of the component produce the same results as the hand check.

2. Subtask B: RMDLib.dat wizard
An automated test suite will be crated that checks that creating/editing data in the RMDLib.dat file works properly. A chemical will be added to RMDLib.dat. Decay products and factors will be added to the new chemical. The structure of the RMDLib.dat file for this entry will be checked. Then the new chemical will be edited and the structure checked once again. Finally the chemical will be deleted.

3. Subtask C: Removal of progeny inventories from the source.
An automated test case will be produced that crates the input file in the STRM and Known Source modules and checks that 1) the module does not ask for the progeny inventories and 2) that a 0 is placed in the existing progeny inventory location.

4. Subtask D: Additional flavors of files.
An automated test will be developed to see that the DES editor allows the user to add the 'radonly', 'chemonly' and 'radchem' string to the 'flavor' of the file. Another test will show that the 'straight' or 'branching' string is added properly to the 'flavor'.

A FRAMES 1.x simulation editor test will be constructed that will test that module that have different 'flavors' are not allowed to connect. This will only be crated if a test case does not already exist.

5. Subtask E: Collapse of the progeny.
Testing of the new ContSel will be done by writing a AutoIt test script that will add a two parents that have progeny (Sr-90 and U238 for example) to a simulation file. The resulting GID should contain the appropriate progeny. Then ConSel will be run again and the Sr-90 will be deleted. The result should be the U238 chain and Y-90. Remember Y-90 is left because it is not clear whether the user intended it separately or not.

The test case for the post processor will be constructed in the following manner. A current FRAMES 1.x simulation that uses progeny will be selected. The gw-case1 might suffice. The AFF, WFF, WCF, SCF, ATO, EPF, RIF, and HIF for this simulation will be captured. A batch file will be constructed that will execute the Progeny-Adjust post-processor on each of these files. The first time the processor is run the output file will be checked by hand calculation to insure that the results are correct. If the case does not have some features (such as a difference in time reporting) the files will be augmented by hand to force this calculation. An automated test script for each file time will then be constructed that will test that the post processor is working properly. This will allow for fast automated testing of this tool in the future. Remember a specific document that describes the automated testing steps and results will be automatically generated from the test script.
